# クレーム調査レポート

**claim id**: C20260614-001
**実施日**: 2026-06-14
**対象機能**: execution
**緊急度推定**: medium

## 1. クレーム原文

```
活動 A を一時停止して、同じ活動を再開せずに次の活動 B に移行したとき、
中断していた時間も活動 A の時間として計時されている。
中断時間が差し引かれるようにする。
活動が開始時間と終了時間を 1 セットしか持っていないとこういう挙動になるだろう。
中断できるようにするには複数の period を持てるようにする。
何度も中断する可能性もあるので、持てる period の数を制限しない。
1:N で period を持てる必要がある。
活動画面の開始時間には最初の開始時間を表示するという妥協で良い。
period をたくさん表示すると煩雑になるから。
```

## 2. 分解結果

### 2.1 期待挙動 (Expected)
- (A) 活動の一時停止中（中断）の時間は、その活動の経過時間から差し引かれる（中断していた区間は計時されない）。
- (B) 「一時停止 → 同じ活動を再開せず次の活動へ移行」した場合も、中断区間が経過から除外される。
- (C) 何度でも中断できるよう、1 活動が複数の計時区間（period）を 1:N・無制限に持てる。
- (D) 活動画面の開始時刻表示は最初の開始時刻のみ（妥協。period を全部出すと煩雑なため）。

### 2.2 現実挙動 (Actual)
- 活動 A を一時停止（pause）したまま、同じ活動を再開（resumeSame）せず「次の活動へ」または「セット終了」へ進むと、中断区間（pause 開始 → 切替時刻）が活動 A の経過時間に混入する。
- `pausedTotalSec` は `resumeSame` でしか加算されないため、「pause したまま次へ」では加算されず 0 のまま `endCurrentItem` が `elapsed = now - startedAt` を確定する。
- 記録方式は startedAt/endedAt の単一ペア + `pausedTotalSec` で、複数回中断を区間単位で表現できない。

### 2.3 発生条件
- 操作手順: 活動 A 計時中 → 一時停止 → （同じ活動を再開せず）「次の活動へ」または「セット終了」。
- 環境: アプリ全般（特定環境依存ではなく実装上の常時再現）。
- 時刻: pause したまま活動切替/終了した時。
- ユーザー属性: 全ユーザー。

### 2.4 影響範囲
- 該当ユーザー: 全ユーザー（execution で pause を使う全員）。
- 業務影響: 中断時間が活動時間に上乗せされ、記録される経過時間が実際より長くなる（データ表示の正確性欠陥）。データ破損（既存履歴の壊れ）はない。
- データ影響: 確定 `elapsedSec` が過大計上されるが、保存形式自体は壊れない。

### 2.5 報告経路
- 経路: 直接（オーナー seiji による動作確認フィードバック）。
- 温度感: 冷静（事象 + remedy 設計案を併記）。

### 2.6 報告者文脈
中断を挟みながら活動したいが、中断中の時間まで活動時間に入ってしまうと記録が実態とずれる。複数回中断しても正しく差し引けるよう、1 活動が複数の計時区間を持てる構造にしたい。開始時刻表示は煩雑さ回避のため最初の 1 つだけで良い。
