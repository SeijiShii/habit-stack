## 1. 変更 UC シナリオ

### UC-EX-RESUME: タブ切替/リロードで計時継続
| シナリオ ID | 前提 | 操作ステップ | 期待結果 |
|---|---|---|---|
| E-RESUME-01 | セット計時を開始し 90 秒経過 | (1) タブを離れる/リロード (2) 実行画面を再表示 | 「開始」に戻らず計時画面が継続。経過は ~90秒+（`startedAt`起点で再算出）。データ消失なし |
| E-RESUME-02 | 2活動目を計時中（1活動目は確定済） | リロード | index/records 復元、2活動目が継続、1活動目の確定経過は保持 |
| E-RESUME-03 | paused 状態でリロード | リロード | paused 復元、経過は pause 時点で凍結のまま |

### UC-EX-IDLE: 4H 放置の自動終了
| シナリオ ID | 前提 | 操作ステップ | 期待結果 |
|---|---|---|---|
| E-IDLE-01 | 計時中、`lastSavedAt` を 4H+ 前に固定（now 注入 or ハートビート古い値を seed） | 実行画面マウント | 活動・セットが `lastSavedAt` で終了（status=done）。ライブタイマー非表示、サマリ導線表示。達成記録される（1活動以上実行） |
| E-IDLE-02 | 計時中、gap=3H59m | マウント | 自動終了せず継続復元。1活動の表示は 4H クランプ範囲内 |

### UC-EX-CAP: 4H キャップ表示
| シナリオ ID | 前提 | 操作ステップ | 期待結果 |
|---|---|---|---|
| E-CAP-01 | 1活動を `startedAt` 5H前で seed し継続復元（gap<4H 条件を満たすケース） | 表示確認 | 経過表示が 4:00:00 相当でクランプ（4H 超表示しない） |

### UC-EX-PERSIST: 15秒バックエンド永続
| シナリオ ID | 前提 | 操作ステップ | 期待結果 |
|---|---|---|---|
| E-PERSIST-01 | 認証済（owner 確立）で計時中 | 15秒以上計時継続 | `/api/sync/push` が進行中 session を `last_saved_at` 付きで送信（network 観測 or サーバ行確認） |

## 2. リグレッションシナリオ（既存 UC、重要度高）

| UC | シナリオ ID | 確認観点 |
|---|---|---|
| 実行フロー（開始→次の活動へ→セット終了） | E-REG-01 | 既存の遷移・確定経過・「次の活動へ」表記が不変（R20260610-001 を壊さない） |
| 背景化/リロード経過（既存 004） | E-REG-02 | 既存の経過維持挙動が回帰しない（今回の復元で強化される方向） |
| offline | E-REG-03 | オフラインでも計時・localStorage 復元が機能、復帰後 outbox 同期 |
| 穴あき達成記録 | E-REG-04 | 1活動以上で achieved=true が自動終了経路でも成立 |

## 3. 移行検証シナリオ（マイグレーションある時）

| シナリオ ID | 移行前データ | 移行後期待状態 |
|---|---|---|
| E-MIG-01 | `last_saved_at` 列追加前の既存 session 行 | 列追加後も既存行は NULL で読み書き可、アプリ正常動作（旧データ非破壊） |
| E-MIG-02 | 旧クライアント（lastSavedAt 未送信）の push | server 行は last_saved_at=NULL、エラーなし |

## 4. 環境要件差分

| 項目 | 前回 | 今回 | 理由 |
|---|---|---|---|
| 時刻注入 | 一部 | 復元/4H 境界で `now` 注入必須 | 決定的な gap/cap 検証 |
| localStorage seed | 不要 | ハートビート seed ヘルパ | 放置/owner 分離シナリオ |
| DB migration | 既存 0000 | 0001 適用済の staging | E-PERSIST/E-MIG |
