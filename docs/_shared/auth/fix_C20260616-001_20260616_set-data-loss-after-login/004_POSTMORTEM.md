> **発生日**: 2026-06-15〜16（R20260615-001 デプロイ後〜翌朝発覚）
> **対応完了日**: （修正リリースで更新）

## 1. 概要
2026-06-15 にデプロイした R20260615-001（account-switch-stop-sync）が、既存アカウントへのサインイン時に
「current owner 以外のローカルデータを物理削除」する cleanup（`wipeOtherOwners`）を導入。userId churn で旧 owner に
取り残されたゲスト作成データが、**未送信 outbox ごと削除**され、ユーザーが「Google ログインしただけ」と認識する操作で
活動セット 1 件＋実績が恒久的に失われた（パーシャル消失）。

## 4. 検知の経緯
- 開発者（報告者）自身のドッグフーディングで翌朝発見。
- 検知遅延の原因: **スモークテストがゲスト→既存アカウント churn × 未同期データの組合せを踏まない**ため、
  デプロイ前後の自動/手動確認で露見しなかった（owner-scoped read で見かけ上は正常）。

## 5. 対応の流れ
1. 検知（ユーザー報告） → /flow:claim 三項照合でバグ確定。
2. 暫定回避策: 有効なユーザー側回避策なし。運用止血 = `wipeOtherOwners` 呼び出し撤去（read 隔離で正当性不要）。
3. 根本修正: owner churn 時のローカルデータ保全（reassign/同期）+ 不可逆操作前確認 + 診断 instrumentation。
4. リリース: 即時 mitigation を前倒し → 恒久修正。
5. 監視継続: 診断ログで実機 owner 遷移を確証後にログ削除。

## 6. 直接原因 + 根本原因
（`001_ROOT_CAUSE.md` から引用）
- 直接原因: `repos.ts:49-53` → `localStore.ts:181-192`(`wipeOtherOwners`) → `:146-171`(`wipeOwner` が entity + 未送信 outbox を削除)。owner churn は `AuthProvider.tsx:91-120,181`。
- 根本原因: R20260615-001 が「正当性に不要な物理 wipe」を、同期保証・ローカル owner 移行・ユーザー意図（連携=保持 vs 上書き）と切り離して導入したこと。

## 7. 学習事項

### 7.1 良かった点
- owner-scoped read（`getAllByOwner`）という設計のおかげで、**物理 wipe は正当性に不要**＝止血が機能リグレッションゼロで可能だった。
- claim→fix の triage が「パーシャル消失」という観察から owner churn 仮説へ短時間で収束。
- DB schema 不変・migration なしのため、コード revert で完全復旧可能。

### 7.2 改善点
- **「再 pull で復元可能」という安全性主張が、サーバ到達済みデータにしか成立しない前提を明文化していなかった**
  （R20260615-001 §4）。未同期データを破壊し得る設計を「可逆」と誤認した。
- `reassignOwner` を「実装済（案B）」と SoT（[論点-009]）に記載しながら**実際は未配線（dead code）**。
  SoT とコードの乖離が owner 移行欠落の見落としを誘発。
- 物理削除（不可逆）を導入する変更で、**「削除対象が未同期＝復元不能になり得るか」のレビュー観点が欠けていた**。
