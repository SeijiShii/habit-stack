## 1. 症状
ログイン状態で活動セット等を作成・記録した後、しばらく（数時間〜日跨ぎ）放置して再訪し、
内部の認証トークン（Clerk セッション）が陳腐化した状態で再読み込みが起きると、**作成済み
データがデバイスから消える（画面に出ない）**。**再ログインしても復旧しない**。

## 2. 期待挙動
- local-first（IndexedDB 正本）で作成したデータは、認証トークンの陳腐化・再認証をまたいでも
  **保持され可視であり続ける**（concept §1.1 UC8 / §3 NFR データ整合性「欠損なく引き継ぐ」）。
- 万一不可視化しても再ログインで復旧する。

## 3. 再現手順（推定・機序ベース）
1. アプリ初回起動 → 匿名ゲストセッション確立（サーバが userId=U1 を発行）。
2. データ作成・計時・記録（U1 所有で IndexedDB に保存）。短時間は再読込でも生存（owner=U1 継続）。
3. Clerk セッションが TTL 失効するまで放置（inactivity timeout / セッション失効）。
4. 再読み込み → `useAuth()` が `isSignedIn:false / userId:null`。
5. 自動ゲスト生成 effect が `/api/auth/guest` を叩く → 失効でサーバセッション無し → **新 userId=U2 を発行**。
6. owner=U2 に切替 → `getAllByOwner(U2)` が空 → **U1 所有データが orphan 化（消失）**。
7. 再ログイン: 匿名運用は U1 再到達不能／連携済みでも reassign 不発で復旧せず。

## 4. 再現環境
| 項目 | 値 |
|---|---|
| ブラウザ | PWA / モバイル・デスクトップブラウザ（端末非依存、Clerk セッション挙動に依存） |
| OS | 非依存 |
| アプリバージョン | 本番（C20260616-001 fix デプロイ後、`wipeOtherOwners` 撤去済） |
| データ条件 | 匿名ゲスト or Google 連携ユーザーが作成したローカルデータが存在 |
| トリガー | Clerk セッションの TTL 失効後の再読み込み（受動・不可避） |

## 5. 影響範囲
- 影響ユーザー数: 報告 1 名。機序上、**セッションを失効させる全ユーザーに潜在波及**（受動トリガー）。
- 影響機能: 全データ（活動セット / セッション / 記録 / 達成）— owner スコープ read 全般。
- データへの影響: **あり**（物理削除ではなく owner 不一致による orphan 化。匿名は事実上不可逆）。
- セキュリティ影響: なし（消失方向。SEC-001 のサーバ owner 強制は不変）。

## 6. 暫定回避策
- ユーザー側: なし（受動トリガーのため回避不能）。
- 運用側（緩和）: Clerk セッション lifetime を延長すると churn 頻度は下がるが**根絶しない**（いずれ失効）。
