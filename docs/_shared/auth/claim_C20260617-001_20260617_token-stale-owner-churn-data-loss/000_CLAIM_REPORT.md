# クレーム調査レポート

**claim id**: C20260617-001
**実施日**: 2026-06-17
**対象機能**: _shared/auth（匿名ゲスト owner 継続性。実体は _shared/local-sync と協働）
**緊急度推定**: high（ユーザーデータの不可逆消失クラス。発火条件＝セッション陳腐化は全ユーザーが時間経過で必ず踏む受動トリガー）

## 1. クレーム原文

```
データが削除されてしまうこと。
ログインしてデータを作成する。その時点では再読み込みしてもデータが生きている。
時間がたち内部で認証トークンが陳腐化し、再読み込みが発生するとデータがデバイスから消える。
再ログインしても保存したはずのデータは復旧しない。
Googleログインはbousai-bag-checkerでも実装しているが、そちらでは問題になっていない。
```

## 2. 分解結果

### 2.1 期待挙動 (Expected)
ログインしてデータを作成したら、時間が経って認証トークンが期限切れになり再読み込みが
発生しても、作成済みデータは**デバイス上に保持され続ける**こと。トークンの陳腐化は
内部的な再認証で透過的に処理され、ユーザーのデータ可視性に影響しない。万一見えなくなっても
**再ログインすれば復旧する**こと（local-first + ログイン＝引き継ぎの設計趣旨どおり）。

### 2.2 現実挙動 (Actual)
- 作成直後〜短時間: 再読み込みしてもデータは生きている（同一セッション継続中＝owner 不変）。
- 時間経過で内部の認証トークン（Clerk セッション）が陳腐化 → その状態で再読み込みが起きると
  **データがデバイスから消える**（画面から消失）。
- **再ログインしても復旧しない**（保存したはずのデータが戻らない）。
= パーシャル/全体を問わず、owner スコープから外れたデータが不可視化する**消失クラス**。

### 2.3 発生条件
- 操作手順: ログイン状態でデータ作成 → 短時間内は再読み込みで生存確認 →（数時間〜日をまたぐ等）
  時間経過で Clerk セッションが期限切れ → その後の再読み込みで消失。
- 環境: 本番（C20260616-001 fix デプロイ済 = `wipeOtherOwners` 撤去後）。PWA/ブラウザ。
- 時刻: セッション TTL 経過後の任意の再読み込み。
- ユーザー属性: 匿名ゲスト〜Google 連携ユーザー双方が対象（後述の機序は両方で起こりうる）。
- **トリガーは受動的かつ不可避**（セッション期限切れは全ユーザーが時間経過で必ず到達）。

### 2.4 影響範囲
- 該当ユーザー: 報告者 1 名。ただし機序上、**セッションを期限切れにする全ユーザーに潜在波及**
  （アプリを一定時間開かない → 期限切れ → 次回起動で消失）。能動操作不要のため latent 影響は広い。
- 業務影響: 継続記録というプロダクト中核価値の信頼性毀損。
- データ影響: 物理削除ではなく **owner 不一致による orphan（不可視）化**。匿名のままなら旧 owner（旧ゲスト
  userId）は再到達不能＝事実上の不可逆消失。Google 連携済みでもサインイン経路で reassign が
  発火しなければ復旧しない（「再ログインしても戻らない」と整合）。

### 2.5 報告経路
- 経路: 開発者（seiji）自身のドッグフーディングからの直接報告。
- 温度感: 冷静・分析的（「内部で認証トークンが陳腐化」「再読み込み」と機序の仮説を添えて報告）。

### 2.6 報告者文脈
報告者が達成したかったこと: 習慣の継続記録。ログインまでして作ったデータが、放置後の再訪で
消えており、再ログインでも戻らない＝記録の継続が破綻し、アプリへの信頼そのものが揺らぐ。
